MPA Biometric Verification System - APK Auto-Build System Technical Documentation

Document Version: 1.0 | Date: February 2026 | Project: exam-biometric-system

Question 1: Where is the Android template stored on the server?

The Android project is generated dynamically per build. Each time admin clicks Build APK, a fresh complete Android project is created at:

/apk-builds/{buildId}/

This path is defined in server/apk-builder.ts:

const APK_BUILDS_DIR = path.join(process.cwd(), "apk-builds");

For example, build ID 5 creates the following structure:

/apk-builds/5/
  build.gradle                            (Project-level Gradle)
  settings.gradle                         (Project settings)
  gradle.properties                       (Gradle JVM + Android config)
  gradle/wrapper/
    gradle-wrapper.properties             (Gradle 8.4 distribution URL)
  keystore/                               (Signing keystore location)
  app/
    build.gradle                          (App-level Gradle with all dependencies)
    proguard-rules.pro                    (ProGuard rules for Mantra SDK, TFLite)
    libs/                                 (Place Mantra MFS100 .aar here)
    src/main/
      AndroidManifest.xml                 (All permissions + activities declared)
      assets/
        config.json                       (Exam config - auto-injected)
      java/com/mpa/verify/{examname}/
        MpaApplication.kt                (Application class)
        model/
          Models.kt                       (Candidate, LoginRequest, etc.)
        network/
          ApiService.kt                   (Retrofit API interface)
          RetrofitClient.kt               (Retrofit setup with server URL)
        db/
          AppDatabase.kt                  (Room DB - candidates + pending)
        ui/
          SplashActivity.kt               (Splash screen + auto-login check)
          LoginActivity.kt                (Operator login)
          DashboardActivity.kt            (Stats + sync + navigation)
          CandidateListActivity.kt        (Candidate list + adapter)
          AttendanceActivity.kt           (Mark attendance)
          VerificationActivity.kt         (Face + fingerprint + OMR)
        service/
          HeartbeatService.kt             (30s heartbeat to server)
          SyncService.kt                  (Upload pending verifications)
        receiver/
          BootReceiver.kt                 (Auto-start on device boot)
      res/
        layout/ (7 XML layouts)
        values/ (strings.xml, themes.xml, colors.xml)
        drawable/ (ic_fingerprint.xml, card_bg.xml)
Note: The package name follows the pattern com.mpa.verify.{examname} where examname is the exam title with non-alphanumeric characters removed and lowercased.

Question 2: Does the generated Android project include Gradle wrapper?

The generated project includes Gradle configuration files but not the binary wrapper files:

FileGenerated?Notes
build.gradle (project level)YesAGP 8.2.2, Kotlin 1.9.22
app/build.gradle (app level)YesAll dependencies, signing config, SDK versions
settings.gradleYesPlugin management + module include
gradle.propertiesYesJVM args, AndroidX, parallel builds
gradle/wrapper/gradle-wrapper.propertiesYesPoints to Gradle 8.4 distribution
gradlew (Unix shell script)NoBinary - must be copied or generated
gradlew.bat (Windows batch)NoBinary - must be copied or generated
gradle/wrapper/gradle-wrapper.jarNoBinary JAR - must be copied or generated

How to add the missing wrapper files:

Option A: Run this once on the server (requires Gradle installed):

cd /apk-builds/{buildId}
gradle wrapper --gradle-version 8.4

Option B: Copy gradlew, gradlew.bat, and gradle/wrapper/gradle-wrapper.jar from any existing Android Studio project into the build directory.

Fallback behavior: The build system checks for gradlew first. If not found, it tries the system gradle command. If neither is available, it creates a downloadable archive (.tar.gz) instead.

Question 3: Where is the release signing keystore stored and how is APK signing configured?

Keystore Location:

/keystore/mpa-release.jks

Relative to each build directory: ../keystore/mpa-release.jks

Gradle Signing Configuration (in app/build.gradle):

signingConfigs {
    release {
        storeFile file('../keystore/mpa-release.jks')
        storePassword System.getenv('KEYSTORE_PASSWORD') ?: 'mpa@2024secure'
        keyAlias 'mpa-verify'
        keyPassword System.getenv('KEY_PASSWORD') ?: 'mpa@2024secure'
    }
}

buildTypes {
    release {
        minifyEnabled true
        proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
                      'proguard-rules.pro'
        signingConfig signingConfigs.release
    }
}

Signing Parameters:

ParameterValue
Keystore file../keystore/mpa-release.jks
Store passwordEnv: KEYSTORE_PASSWORD (default: mpa@2024secure)
Key aliasmpa-verify
Key passwordEnv: KEY_PASSWORD (default: mpa@2024secure)

How to create the keystore (run once on server):

mkdir -p keystore

keytool -genkey -v \
  -keystore keystore/mpa-release.jks \
  -alias mpa-verify \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -storepass mpa@2024secure \
  -keypass mpa@2024secure \
  -dname "CN=MPA Verification, O=MPA, L=India, C=IN"
Important: For production, change the default passwords and set them via environment variables KEYSTORE_PASSWORD and KEY_PASSWORD on your server.

Question 4: What exact command is executed during /api/apk/build to generate the APK?

The build is triggered by POST /api/apk/build/:examId. The exact command executed (from server/apk-builder.ts):

Primary command (if gradlew exists):

cd "/apk-builds/{buildId}" && ./gradlew assembleRelease --no-daemon 2>&1

Fallback command (if gradlew not found, system gradle used):

cd "/apk-builds/{buildId}" && gradle assembleRelease --no-daemon 2>&1

Command parameters:

ParameterValue
Working directory/apk-builds/{buildId}
Timeout300,000 ms (5 minutes)
Max output buffer50 MB
Flags--no-daemon (prevents background daemon)
Output redirect2>&1 (stderr merged into stdout for logging)

Complete build flow (6 steps):

StepProgressAction
Step 110%Create project directory
Step 220%Generate all Android project files (Kotlin, XML, Gradle)
Step 330%Inject exam config into assets/config.json
Step 440%Check for Android SDK (ANDROID_HOME env variable)
Step 560%Run Gradle assembleRelease (or create ZIP archive if no SDK)
Step 690-100%Locate APK, copy to /public/apks/, update database
If ANDROID_HOME is not set: The system skips Gradle entirely and creates a .tar.gz archive of the project that can be downloaded and opened in Android Studio for manual compilation.

Required environment for auto-build on server:

# Install JDK
apt install openjdk-17-jdk

# Install Android SDK command-line tools and set environment variable
export ANDROID_HOME=/path/to/android-sdk

# Install required SDK components
sdkmanager "build-tools;34.0.0" "platforms;android-34"

Question 5: Where are the built APK files stored publicly for download?

Output directory:

/public/apks/

Defined in server/apk-builder.ts:

const APK_OUTPUT_DIR = path.join(process.cwd(), "public", "apks");

File naming pattern:

ScenarioFile name patternExample
With Android SDK (APK built)MPA_Verify_{ExamName}_v{version}.apkMPA_Verify_SSCExam_v3.2.14.apk
Without Android SDK (archive)MPA_Verify_{ExamName}_v{version}.zipMPA_Verify_RailwayExam_v3.1.7.zip

Download methods:

Method 1: API endpoint

GET /api/apk/download/:examId

This finds the latest successful build for the specified exam and streams the file with proper headers:

  • APK: Content-Type: application/vnd.android.package-archive
  • Archive: Content-Type: application/gzip
  • Header: Content-Disposition: attachment; filename=...

Method 2: Direct static file URL

https://mpaverification.com/public/apks/MPA_Verify_SSCExam_v3.2.14.apk

The directory is served as static files via Express:

app.use('/public/apks', express.static(apkOutputDir));

Full directory example after multiple builds:

/public/apks/
  MPA_Verify_SSCExam_v3.2.14.apk
  MPA_Verify_RailwayExam_v3.1.7.zip
  MPA_Verify_UPSCPrelims_v3.0.5.apk
  MPA_Verify_StatePSC_v3.3.42.zip
Admin Panel Integration: After a successful build, the "Download APK" button (purple) appears in the APK management page. The button calls /api/apk/download/:examId which handles finding and streaming the correct file.
